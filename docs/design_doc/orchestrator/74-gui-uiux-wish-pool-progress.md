# DD-074: GUI 交互设计 — 许愿池 + 进度观察 + 专家模式

## 概述

定义 orchestrator GUI 的用户界面交互设计，面向非工程师用户。两个一级页面（许愿池、进度观察）覆盖核心流程，专家模式在二级页面以下提供完整工作流可见性。本文档是 FR-064 的详细设计。

## 设计决策

### 1. 用户画像与场景

**主要用户**：非工程师（产品经理、项目负责人、业务分析师）

| 特征 | 描述 |
|------|------|
| 技术水平 | 不了解 CLI、YAML、工作流概念 |
| 核心需求 | 用自然语言提需求、看进度 |
| 使用频率 | 每天 1-3 次（提需求/查进度） |
| 耐心阈值 | 5 秒内无反馈会焦虑 |
| 平台 | macOS 为主，Windows 次之 |

**关键用户旅程**：
```
打开 GUI → 输入需求 → 等待 FR 草稿 → 审阅 → 确认开发 → 切换到进度观察 → 看到实时进度
```

### 2. 导航结构

采用**顶部 Tab 切换**而非侧边栏，理由：
- 仅 2 个一级页面，侧边栏浪费空间
- Tab 模式用户认知负荷更低
- 与移动端底部 Tab 体验一致

```
┌────────────────────────────────────┐
│  🌟 许愿池    📊 进度观察    [设置] │  ← 固定顶栏
├────────────────────────────────────┤
│                                    │
│         页面内容区                   │
│                                    │
└────────────────────────────────────┘
```

**页面导航深度**：

| 层级 | 许愿池 | 进度观察 |
|------|--------|---------|
| L1 | 许愿列表 + 输入区 | 任务卡片列表 |
| L2 | 许愿详情（FR 草稿预览） | 任务详情（步骤 + 日志） |
| L3 | — | 专家模式（工作流/资源/Agent） |

L2/L3 页面左上角始终有"← 返回"按钮，点击回到上一层。

### 3. 许愿池交互设计

#### 3.1 输入体验

- **大文本区**：最小高度 120px，自动扩展，Liquid Glass 卡片样式
- **占位文字**："描述你想要实现的功能，比如：我想让用户能通过邮箱注册账号..."
- **字符限制**：2000 字（与 TaskCreate goal 字段对齐），显示剩余字数
- **快捷操作**：Cmd/Ctrl+Enter 提交

#### 3.2 等待体验

许愿提交后，用户最关注"系统在做什么"。设计要点：

1. **即时反馈**（0-1s）：按钮变为"许愿已发送"，输入区渐隐
2. **进度提示**（1s-完成）：
   - "正在理解你的需求..."
   - "正在设计功能方案..."
   - "正在撰写 FR 文档..."
   - （文字随 `TaskFollow` 的 step 变化更新）
3. **完成提示**：卡片从"草稿中"变为"待确认"，自动展开预览

#### 3.3 FR 草稿预览

- Markdown 渲染（使用 `react-markdown` 或类似库）
- 只读模式，不可直接编辑草稿文本
- 底部三个操作按钮：
  - **确认开发**（主按钮，`--accent`）→ 触发开发 workflow，跳转进度观察
  - **修改需求**（次要按钮）→ 回到输入区，预填原始需求文本
  - **取消**（文字按钮，`--danger`）→ 确认对话框 → TaskDelete

#### 3.4 许愿列表

- 卡片式列表，按时间倒序
- 每张卡片显示：状态标签 + 需求摘要（前 50 字）+ 时间
- 状态筛选：全部 / 草稿中 / 待确认 / 已确认 / 已取消
- 空状态："还没有许过愿，在上方输入你的第一个需求吧"

### 4. 进度观察交互设计

#### 4.1 任务卡片

每张卡片信息密度适中，非技术用户能快速判断状态：

```
┌─────────────────────────────────┐
│ 🔵 运行中                        │  ← 状态标签（颜色 + 图标 + 文字）
│ 用户登录功能                      │  ← 任务名称
│ ████████░░░░░░░░  3/8            │  ← 进度条 + 完成数
│ 开始于 10:30         预计还需 15m │  ← 时间信息
└─────────────────────────────────┘
```

**排序**：运行中 > 暂停 > 失败 > 已完成（同状态内按时间倒序）

**自动刷新**：`TaskWatch` streaming 实时推送状态变更，无需手动刷新。

#### 4.2 任务详情页

**上半区**：步骤进度
- Item 列表，每条显示：序号 + 名称 + 状态图标 + 耗时
- 当前运行中的 Item 高亮显示
- 已完成 Item 可折叠

**下半区**：实时日志
- `TaskFollow` streaming 持续追加
- 自动滚动到底部（可暂停自动滚动）
- 日志条目带时间戳前缀
- 最多显示最近 500 条（更早的可滚动加载）

**操作栏**（固定在详情页顶部右侧）：
- **暂停**（running 状态显示）/ **恢复**（paused 状态显示）
- **重试**（仅 failed 状态显示）
- **专家**（toggle 按钮，进入/退出专家模式）

### 5. 专家模式交互设计

#### 5.1 进入方式

- 任务详情页右上角"专家"按钮
- 点击后：详情页下半区（日志区）被专家 Tab 面板替换
- 日志仍可在专家 Tab 中的子页查看
- 按钮变为"专家 ✓"表示已开启

#### 5.2 专家 Tab 设计

```
[工作流]  [资源]  [Agent]  [日志]  [原始数据]
```

**工作流 Tab**：
```
┌────────────────────────────────┐
│  [需求分析] → [架构设计] → ... │  ← DAG 横向流程图
│      ✅          ✅             │
│                                │
│  → [前端开发] → [后端开发]      │
│       ✅          🔵           │
│                                │
│  → [测试] → [部署]             │
│      ⬜      ⬜               │
└────────────────────────────────┘
```
- 使用 SVG 绘制简化 DAG（节点 + 箭头）
- 点击节点高亮对应 Item 的日志段
- 数据来源：`TaskInfo` 中的 `graph_debug` 字段

**资源 Tab**：
```
┌────────────────────────────────┐
│  类型筛选: [全部] [Workflow]    │
│           [Agent] [StepTemplate]│
├────────────────────────────────┤
│  📄 dev-workflow    Workflow    │
│  🤖 code-agent      Agent      │
│  📋 test-template   StepTmpl   │
├────────────────────────────────┤
│  已选: dev-workflow             │
│  ┌──────────────────────────┐  │
│  │ apiVersion: orchestrator │  │
│  │ kind: Workflow           │  │
│  │ metadata:                │  │
│  │   name: dev-workflow     │  │
│  │ spec:                    │  │
│  │   steps: ...             │  │
│  │              [编辑] [复制]│  │
│  └──────────────────────────┘  │
└────────────────────────────────┘
```
- 左侧资源列表（`ResourceGet`），右侧详情（`ResourceDescribe`）
- 编辑按钮（需 operator 角色）→ 打开 YAML 编辑器 → 提交（`ResourceApply`）
- 只读模式下编辑按钮灰显

**Agent Tab**：
```
┌──────────────────────────────────────┐
│  名称          状态     健康    任务  │
├──────────────────────────────────────┤
│  code-agent    active   ✅     3    │
│  test-agent    cordoned ⚠️     0    │
│  review-agent  draining 🔴     1    │
├──────────────────────────────────────┤
│  [Uncordon]  [Drain]   ← admin only │
└──────────────────────────────────────┘
```
- 数据来源：`AgentList` RPC
- 操作按钮仅 admin 角色可见
- Cordon/Uncordon 无需确认对话框
- Drain 需确认对话框（不可逆操作）

**日志 Tab**：
- 与非专家模式下的日志流相同
- 额外功能：日志级别筛选、关键字搜索

**原始数据 Tab**：
- 完整 `TaskInfo` JSON，语法高亮
- 可折叠的树形视图
- 复制按钮（复制完整 JSON 到剪贴板）

### 6. 状态流转与页面联动

#### 6.1 许愿 → 进度 联动

```
许愿池: 用户确认 FR
  ↓ TaskCreate(workflow="development")
  ↓ 自动跳转
进度观察: 新任务出现在列表顶部（运行中状态）
```

- 跳转后 Tab 自动切换到"进度观察"
- 新任务卡片有短暂高亮动画（引导用户注意）

#### 6.2 状态通知

当 GUI 在后台运行时，通过系统通知告知关键状态变更：
- 任务完成："用户登录功能 开发完成"
- 任务失败："自动发送周报 执行失败，点击查看详情"
- FR 草稿就绪："你的需求方案已生成，等待确认"

通知通过 Tauri notification API 实现。

### 7. 错误处理与边界场景

| 场景 | 处理方式 |
|------|---------|
| daemon 未启动 | 显示连接向导，引导启动或配置 |
| daemon 断开连接 | 顶栏显示"连接中断"横幅，自动重连（3 次，1s 间隔） |
| gRPC 调用失败 | toast 提示错误信息（非技术语言），如"操作失败，请稍后重试" |
| RBAC 权限不足 | 按钮灰显 + tooltip "需要更高权限" |
| 网络超时 | 15s 超时，显示"网络超时"提示 + 重试按钮 |
| 空状态 | 各列表显示友好的空状态插图 + 引导文字 |

### 8. 键盘快捷键

| 快捷键 | 操作 |
|--------|------|
| `Cmd/Ctrl + 1` | 切换到许愿池 |
| `Cmd/Ctrl + 2` | 切换到进度观察 |
| `Cmd/Ctrl + Enter` | 提交许愿 / 确认 FR |
| `Escape` | 返回上一层 / 关闭专家模式 |
| `Cmd/Ctrl + E` | 切换专家模式（在任务详情页） |
