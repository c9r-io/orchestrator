# QA Docs

This directory contains reproducible, verifiable QA test documents. They are executed by `qa-testing`, which creates `docs/ticket/` tickets on failures.

Goals:
- Any tester can reproduce each QA doc step-by-step.
- Steps and assertions are explicit. Include runnable API/DB verification commands when needed.
- The document structure is stable, making it easier for agents to automate execution and compare expected results.

## Suggested Directory Structure

Organize by module:

```
docs/qa/
├── README.md
├── <module>/
│   ├── 01-<topic>.md
│   └── 02-<topic>.md
└── ...
```

## Document Rules (Strict)

Keep each document to at most 5 scenarios to enable parallel testing and fast regression.

Each scenario must include:
1. Preconditions (initial state)
2. Steps (reproducible)
3. Expected results (verifiable assertions)
4. Optional: data checks (SQL/query commands)
5. Optional: cleanup/rollback steps

## Recommended Template

```markdown
# <Module>: <Doc Title>

## Scope
- In scope: ...
- Out of scope: ...

## Environment
- UI: <url or route>
- API: <base url>
- Auth:
  - If required: how to obtain token (script/command)
- DB (optional): <dsn / container / cli command>

## Scenario 1: <Title>

### Preconditions
- ...

### Steps
1. ...
2. ...

### Expected
- UI: ...
- API: ...

### DB Checks (optional)
```sql
SELECT ...;
```

## Scenario 2: ...
```

## Local Environment Setup (project-bootstrap Convention)

If the project was generated by `project-bootstrap`, the following entry points typically exist:

```bash
# 1) Start services (Docker Compose)
docker compose -f docker/docker-compose.yml up -d --build

# 2) Optional: clean and reset environment
./scripts/reset-docker.sh

# 3) Optional: deploy/upgrade on K8s
./deploy/deploy.sh
./deploy/upgrade.sh
```

Notes:
- Actual ports, service names, and dependencies are defined by the project's `docker/docker-compose.yml` and `deploy/` scripts.
- If the API requires a token, the project should provide a script/command to obtain it and document it under `Environment` in QA docs.
