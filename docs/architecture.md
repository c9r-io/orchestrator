# Reference Architecture (Platform Template)

This document describes the default architecture conventions and optional extensions for projects generated by `project-bootstrap`. It is a platform-level reference architecture and is not bound to any specific business domain (it does not assume identity, RBAC, OIDC, etc.).

## 1. Project Overview

Platform goals:
- Enable fast iteration by AI and developers under a shared directory layout and workflow.
- Turn `plan -> qa-doc-gen -> qa-testing -> ticket-fix` into an executable, reproducible acceptance loop.
- Provide consistent operational entry points for both local (Docker Compose) and cluster (Kubernetes) environments.

## 2. Directory Layout And Components (project-bootstrap)

Default directory structure:

```
core/                 # Backend service (Rust by default; language/framework can be swapped)
portal/               # Admin portal / Web UI (optional)
docker/               # Dockerfiles + docker-compose.yml
k8s/                  # Kubernetes manifests (base)
deploy/               # deploy/upgrade/cleanup scripts
scripts/              # Local helper scripts (including reset-docker.sh)
docs/qa/              # Reproducible QA docs (input to qa-testing)
docs/design_doc/      # Design docs (captured from plan mode; can be generated by qa-doc-gen)
docs/ticket/          # QA failure tickets (input to ticket-fix)
```

Conventional entry points:
- Local start: `docker compose -f docker/docker-compose.yml up -d --build`
- Local reset: `./scripts/reset-docker.sh`
- Cluster deploy: `./deploy/deploy.sh`
- Cluster rolling upgrade: `./deploy/upgrade.sh`
- Cluster cleanup: `./deploy/cleanup.sh`

## 3. System Architecture (Reference)

### 3.1 Overview

```mermaid
flowchart TB
    subgraph clients[Client Layer]
        Portal[portal<br/>Web UI (optional)]
        External[External Clients (optional)]
    end

    subgraph app[Application Layer]
        Core[core<br/>Backend Service]
    end

    subgraph deps[Dependencies (optional)]
        DB[(Database)]
        Cache[(Cache)]
        MQ[(Queue)]
        IdP[(OIDC Provider)]
    end

    Portal -->|HTTP| Core
    External -->|HTTP/gRPC| Core
    Core --> DB
    Core --> Cache
    Core --> MQ
    Core --> IdP
```

Notes:
- The platform does not require DB/Cache/MQ/IdP; all are optional extensions.
- If needed, encapsulate external dependency boundaries behind mockable interfaces so unit tests do not depend on Docker.

### 3.2 Deployment Model

Local development:
- Docker Compose starts `core/`, `portal/`, and optional dependency services.

Kubernetes:
- `k8s/base` provides the baseline resources for `core/` and `portal/` (Deployment/Service, etc.).
- `deploy/*.sh` provides deploy and rolling-upgrade entry points.

## 4. Suggested Tech Stack (Not Mandatory)

### 4.1 Backend (Rust By Default)

Recommended stack (swap as needed):
- Web: axum + tower
- Async: tokio
- Serialization: serde
- DB: sqlx (or other DB library)
- Logging: tracing
- Testing: tokio + mockall + wiremock (mock external dependencies)

Testing strategy:
- Unit tests should not depend on Docker by default.
- If integration tests require real dependencies, run them in a separate layer and provide deterministic reset/seed mechanisms.

### 4.2 Frontend (React + Vite By Default)

Recommended stack (swap as needed):
- React + TypeScript + Vite
- Unit tests: Vitest + Testing Library
- E2E: Playwright (a small number of critical flows)

## 5. Test Layering And Acceptance Loop (Core Platform Convention)

### 5.1 Three-Layer Test Model

1. Unit tests (most, fastest)
- Focus on business logic and boundary conditions.
- Do not depend on Docker.

2. Integration tests (few)
- Validate external dependency boundaries (DB/Cache/external API).
- Require deterministic environment reset and data setup.

3. E2E (fewest)
- Cover critical user journeys.
- Full-stack E2E must be reproducible; prefer `./scripts/reset-docker.sh` to clean the environment.

### 5.2 Plan To QA Loop

Platform workflow:
1. Developers and AI complete a plan (including acceptance criteria and a test plan).
2. Implement the code.
3. Optional: use `qa-doc-gen` to convert the plan into `docs/design_doc/**` design docs (capture goals/scope/tradeoffs/risks/observability/acceptance criteria).
4. Use `qa-doc-gen` to convert the plan into `docs/qa/**` QA docs (reproducible steps and assertions).
5. Use `qa-testing` to execute QA docs; on failures, create `docs/ticket/**` tickets.
6. Use `ticket-fix` to reproduce/fix/reset/regress and close the ticket after verification.

## 6. Observability And Operations (Recommended)

Minimum baseline:
- `core` exposes `/health` and `/ready` (or equivalent probes).
- Docker/K8s logs are easy to triage (consistent prefixes, structured logs).
- Critical external calls have configurable timeouts and retry policies.

Related skills:
- `ops`: local/cluster status, logs, rollout triage.
- `reset-local-env`: fast reset when the environment is dirty.
- `grpc-regression`: run `grpcurl` from inside the Docker network when host-to-container networking is unreliable.
