# Orchestrator Phase 2 Task 04

## Title

降低 active config 历史漂移的人工修复成本（配置自愈 / 漂移治理）

## Goal

当数据库中的持久化 active config 与仓库 manifest 漂移，或者残留历史脏配置时，减少必须人工 `apply` 才能恢复的场景。

## Problem

这次已经确认过一类真实问题：

- 源 YAML 已经正确
- 但数据库里的 active config 仍保留旧的错误字段组合
- 导致当前命令行为被坏配置影响
- 最终需要人工 re-apply manifest 才恢复

这类问题虽然可以人工修复，但有几个明显缺点：

- 故障定位成本高
- 很像“当前代码坏了”，但根因其实是历史持久化状态
- 对自举系统来说，缺少基本的配置自愈能力

## Scope

- 识别持久化 config 与源 manifest 的常见漂移类型
- 为可安全纠正的历史脏配置增加修复或重建机制
- 降低只因历史状态残留导致的命令异常

## Out Of Scope

- 不做通用数据库迁移框架重写
- 不自动覆盖所有运行配置
- 不在没有明确安全边界的情况下静默重写用户配置

## Acceptance Criteria

1. 常见“历史脏配置残留”不再只能靠人工 re-apply 排查。
2. 系统能区分“源配置本身无效”和“持久化状态已过期/漂移”。
3. 相关 CLI/诊断信息能更快提示用户这是配置漂移，而不是实现 bug。

## Suggested Verification

- 构造一个持久化脏配置但源 manifest 正确的场景
- 验证系统能：
  - 识别漂移
  - 给出明确提示
  - 或通过受控方式修复到可运行状态
