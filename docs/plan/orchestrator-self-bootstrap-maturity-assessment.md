# Orchestrator Self-Bootstrap Maturity Assessment

本文档基于 `resource.rs` 重构任务的完整执行过程，总结当前 orchestrator 的自举 / 自我演化能力边界，并给出下一阶段的治理目标。

---

## 1. 结论

当前 orchestrator 已经具备：

- 可工作的自举能力
- 受控的两轮式自我修正能力
- 基于测试与文档审计的初步闭环能力

但它仍未达到“稳定可信的自我演化系统”标准。

更准确地说，它目前处于：

- **能真实改自己**
- **能在一定程度上检查自己**
- **但还不能完全可靠地治理自己**

---

## 2. 本次任务证明了什么

### A. 已证明具备的能力

1. **真实改造能力**
   - orchestrator 成功驱动了 `core/src/resource.rs` 的拆分重构，而不是停留在演示级别的样例任务。
   - 最终代码通过了 `cargo check`、`cargo test --lib resource`、`cargo test --lib`。

2. **两轮式自举策略可行**
   - 第一轮偏生产，第二轮偏复审 / 收尾。
   - 第二轮能基于第一轮结果继续修正缺口，而不是简单重复执行。

3. **阶段化反馈闭环有效**
   - `self_test`、`align_tests`、`doc_governance` 能对前序产物施加约束。
   - 这使“写代码 -> 验证 -> 收尾”的链路在同一工作流中成立。

4. **文档治理可纳入自举流程**
   - `doc_governance` 实际运行并产出结论，说明系统具备初步的“元治理”能力，而不只是代码流水线。

### B. 已确认的限制

1. **调度语义仍存在脆弱点**
   - `self_test` 一度在 builtin 语义上被错误解释为 capability 匹配，导致任务直接失败。
   - 这说明工作流声明与执行语义之间仍有不一致风险。

2. **观测层可信度不足**
   - `task trace` 持续稳定误报 `overlapping_cycles`。
   - `cycle ended_at` 与 `wall_time_secs` 的重建不完整。
   - 系统能执行，但还不能稳定准确地解释自己的执行过程。

3. **任务创建与 QA 流程耦合过重**
   - 即便禁用 `qa_testing` / `ticket_fix`，`task create` 仍会因缺少 QA markdown 失败。
   - 纯重构任务仍会被 QA 文档前置约束卡住。

4. **运行期控制面不够稳定**
   - 任务运行中，`task info` / `task logs` 的可用性不稳定，监控时需要直接读取 SQLite 与日志目录作为兜底。

5. **plan 阶段存在“低产出存活”现象**
   - 多次出现进程心跳正常、CPU 很低、日志几乎不增长的慢启动窗口。
   - 虽然最终完成，但这会降低可预测性与成本效率。

---

## 3. 当前能力评级

按工程成熟度分层，当前可评为：

- **L1 可脚本化自动执行**: 已超过
- **L2 可完成真实闭环改造**: 已达到
- **L3 可稳定自监督迭代**: 部分达到
- **L4 可高信任持续自演化**: 未达到

建议的当前评级：

- **自举能力**: B+
- **自我演化能力**: B-
- **可运维可信度**: C+
- **观测可信度**: C

---

## 4. 关键判断

本次实践最重要的结论不是“模型会不会写代码”，而是：

- **代码生成与修正能力已经够用**
- **真正限制上限的是 orchestrator 自身的调度、观测、流程解耦与运行期控制面**

换句话说，下一阶段最值得优先投入的，不是继续扩大自举任务范围，而是先提升 orchestrator 自身的基础治理能力。

---

## 5. 下一阶段必补的 5 个基础能力

### 1. 修正 Step 语义一致性

目标：

- 保证 `builtin` / `required_capability` / `step_type` 的解释一致
- 避免声明为 builtin 的步骤被错误落到 agent capability 选择分支

验收信号：

- `self_test` / `ticket_scan` / `loop_guard` / `init_once` 的执行语义在配置校验与运行期保持一致
- 不再需要依赖临时 workflow workaround 才能让 builtin 步骤跑通

### 2. 修正 Trace 重建与异常检测

目标：

- 修复 `overlapping_cycles` 误报
- 为每个 cycle 正确补齐 `ended_at`
- 为 completed task 生成有效的 `wall_time_secs`

验收信号：

- `task trace --json` 中 cycle 边界与事件序列一致
- 正常两轮任务不再被标记为 cycle overlap

### 3. 解耦任务创建与 QA Markdown 前置依赖

目标：

- 允许纯重构 / 纯内核任务在禁用 QA 阶段时正常创建和运行

验收信号：

- 当 workflow 未启用 `qa_testing` / `ticket_fix` 时，`task create` 不强制要求 QA markdown
- QA 依赖应由 workflow 实际启用的阶段驱动，而不是全局硬编码前置条件

### 4. 提升运行期控制面稳定性

目标：

- 提高 `task info` / `task logs` / `watch` 在运行中的可用性
- 降低必须手查 SQLite / `data/logs/` 的频率

验收信号：

- 运行中查询不再随机返回空结果或异常退出
- CLI 输出能稳定反映当前步骤、最近日志与进程状态

### 5. 增加“低产出检测”与计划阶段可观测性

目标：

- 能识别“进程仍存活，但长时间无有效输出增长”的慢速空转状态

验收信号：

- 对 `plan` / `implement` 等长时步骤增加更有解释力的心跳指标
- 可区分“正常长时推理”与“低产出可疑挂起”

---

## 6. 建议的推进顺序

建议先修系统底座，再扩大自举范围：

1. **先修 trace / 调度语义**
   - 先提升系统对自身行为的解释能力

2. **再修 task create / QA 耦合**
   - 让纯重构任务不再被流程噪音阻塞

3. **再修控制面稳定性**
   - 提升运行期观测与运维体验

4. **最后补低产出检测**
   - 优化效率与资源利用，而不是先追求更大规模自动化

---

## 7. 下一阶段目标

下一阶段的主目标应当是：

> 将 orchestrator 从“可工作的自举框架”推进到“可被工程团队信任的自我演化底座”。

这意味着：

- 优先提升系统自解释能力
- 优先提升失败场景下的可恢复性
- 优先降低流程耦合带来的误阻塞

在这些问题没有收口前，不建议将其视为高信任的长期自治系统。
