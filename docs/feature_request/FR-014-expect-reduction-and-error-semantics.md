# FR-014 关键路径 `expect()` 清退与错误语义收敛

- ID: `FR-014`
- Priority: `P1`
- Status: `Proposed`
- Owner: `orchestrator core`

## 背景

仓库已经通过 clippy 约束非测试代码避免 `panic/unwrap/expect`，这是很好的工程纪律。但随着代码增长和模块分化，仍需持续治理“关键路径错误语义是否足够稳定、是否都能被上抛和诊断”这个问题。

本 FR 的重点不是“把测试里的 `expect()` 全部机械替换”，而是确保关键执行链路、控制面链路、启动链路和密钥/配置链路都使用可恢复、可诊断、可审计的错误语义。

## 问题陈述

- 关键路径中若残留 `expect()` 或等价硬失败语义，会放大异常输入或边界条件对进程稳定性的影响。
- 即使没有显式 `expect()`，错误类型过度泛化也会让上游难以区分用户错误、配置错误、系统错误和安全拒绝。
- 目前错误语义仍有继续标准化的空间。

## 目标

- 对关键路径进行 `expect()` 与硬失败语义排查和治理。
- 优先用 `?`、上下文化错误和明确错误类型替换关键路径上的硬终止行为。
- 让调用方能够稳定区分主要错误类别。

## 非目标

- 不要求清退测试代码中的断言式 `expect()`。
- 不要求将所有 `anyhow::Error` 一次性替换为枚举错误。
- 不以“零 `expect()` 文本出现次数”作为目标。

## 范围

### In

- daemon 启动与控制面路径
- 配置加载、验证与持久化路径
- 任务调度与执行主路径
- SecretStore/密钥处理主路径

### Out

- 单元测试内部断言辅助
- 纯示例或一次性脚本

## 需求

### 1. 关键路径清单化治理

- 必须识别并列出关键路径中的硬失败点。
- 对每个治理点明确：
  - 当前失败方式
  - 替换后的上抛语义
  - 调用方可见行为

### 2. 错误类型收敛

- 对以下类别错误应有稳定区分：
  - 用户输入错误
  - 配置校验错误
  - 资源不存在或状态不合法
  - 安全拒绝
  - 外部依赖失败
  - 内部不变量破坏
- 关键边界层不应只返回无差别字符串错误。

### 3. 上下文信息保留

- 任何被上抛的错误必须保留足够上下文，至少能够定位模块、对象和操作。
- 不得为了避免 `expect()` 而牺牲错误可诊断性。

### 4. 回归保护

- 对曾经依赖硬失败语义的路径，必须补足回归测试。
- 关键失败路径应验证“返回受控错误而非进程异常终止”。

## 验收标准

- 关键路径完成一次系统性 `expect()` / 硬失败排查。
- 识别出的关键治理点均已转为可传播、可诊断的错误语义。
- 关键失败路径新增回归测试。
- 文档记录哪些 `expect()` 仅保留在测试或明确不可恢复断言中。

## 风险与缓解

- 风险：为追求统一错误类型引入过度抽象。
  - 缓解：优先在边界层收敛，不强迫全仓库一次性统一。
- 风险：替换后丢失调试信息。
  - 缓解：使用 `context` 和结构化字段保留关键定位信息。
- 风险：行为变化导致旧测试失效。
  - 缓解：先固定预期错误语义，再逐步替换实现。
