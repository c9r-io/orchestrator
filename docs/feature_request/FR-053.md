# FR-053: Full-QA Workflow 大规模 Item 分发中断 — max_cycles_enforced 过早触发

**优先级**: P0
**状态**: Proposed
**日期**: 2026-03-16
**发现场景**: full-qa-execution.md 全量 QA 回归测试

---

## 1. 问题

使用 `full-qa` workflow 执行 131 个 QA 文档的全量回归测试时，scheduler 在仅处理 5 个 item 后即触发 `max_cycles_enforced`，导致剩余 126 个 item 从未被分发执行。

### 复现路径

```
full-qa workflow, 131 items, max_parallel=4, max_cycles=1, mode=fixed
  → cycle 1 started
  → item segment 并行路径: semaphore(4) + JoinSet
  → items 1-4 spawned, 各获得 semaphore permit
  → item 4 (d53a96b9) 完成 exit=0, permit 释放
  → item 5 (ce83c159) 获得 permit, 开始执行
  → 28 秒后, max_cycles_enforced 触发 (current_cycle=1 >= proactive_max=1)
  → inflight_runs_detected: 4 个 command_runs 仍 exit=-1
  → wait_for_inflight_runs 等待 ~5 分钟
  → items_compensated → task_completed (1/131 items)
```

### 根因分析

`execute_item_segment` 的并行路径 (`segment.rs:316-390`) 使用 `Semaphore + JoinSet` 模式：

```rust
for item in items {
    isolation::ensure_item_isolation(...).await?;
    let permit = semaphore.clone().acquire_owned().await?;
    join_set.spawn(async move {
        let _permit = permit;
        process_item_filtered_owned(...).await
    });
}
while let Some(result) = join_set.join_next().await { ... }
```

For-loop 在所有 permit 被占用时阻塞于 `acquire_owned()`。当 item 1-4 各持有一个 permit 并在 JoinSet 中执行时，item 5+ 的迭代被阻塞。**正常情况下，for-loop 应当一直阻塞直到所有 131 个 item 被处理完毕。**

但观测到的行为是：仅 5 个 item 进入了 prehook 评估阶段（`step_prehook_evaluated` 事件），for-loop 似乎在处理完第 5 个 item 后提前退出，导致 `execute_cycle_segments` 返回，cycle loop 进入下一轮迭代并触发 `max_cycles_enforced`。

#### 可能的中断路径

1. **Tokio task 取消**: 若 daemon worker 或 runtime 在执行期间被中断（例如 daemon self-restart 的 exec()），for-loop 会被取消而非返回 Err
2. **SQLite 并发竞争**: 4 个 JoinSet task 并发写入 SQLite（heartbeat、events），在高 WAL 压力下可能导致 busy timeout，引发 `?` 错误传播
3. **Semaphore 异常关闭**: 若 semaphore 被意外 drop（例如 JoinSet panic），`acquire_owned()` 返回 Err
4. **Daemon 进程替换**: `cargo build --release` 期间 daemon 的 self-restart 机制可能触发 exec()，中断正在执行的 worker

### 影响范围

| Workflow | Items | 是否受影响 |
|----------|-------|-----------|
| full-qa | 131 | **是** — 必定触发 |
| self-bootstrap | 2-10 | 不确定 — items 少，可能在中断前完成 |
| 任何 items > max_parallel 的 workflow | N > 4 | 潜在风险 |

## 2. 设计方案

### 2.1 调查阶段（必须先于实现）

由于根因尚未确认为上述 4 个路径中的哪一个，建议先进行以下调查：

1. **添加结构化日志**：在 `execute_item_segment` 的 for-loop 入口和出口添加 `tracing::info!`，记录 items 总数和已处理数
2. **在并行路径的 `?` 传播点添加日志**：`ensure_item_isolation`, `acquire_owned`, JoinSet error collection
3. **排查 daemon self-restart 干扰**：检查 `cargo build --release` 是否触发了 file-watcher 或 self-restart 逻辑
4. **复现**：重新运行 `full-qa` workflow 并观察新增日志

### 2.2 候选修复方案

#### 方案 A: 防御性 for-loop 完成检查

在 `execute_item_segment` 的并行路径结束后，检查实际处理的 item 数是否与预期一致：

```rust
let expected = items.len();
let actual = item_state.len(); // items that went through JoinSet
if actual < expected {
    anyhow::bail!(
        "parallel item segment incomplete: processed {}/{} items",
        actual, expected
    );
}
```

这不能修复根因，但能将问题从静默成功变为显式失败。

#### 方案 B: Cycle 完成屏障

在 `run_task_loop_core` 的 cycle loop 中，在 `max_cycles_enforced` 检查之前，验证所有 item 是否已完成当前 cycle 的 item-scoped steps：

```rust
// 在 max_cycles check 之前
let pending_items = count_items_without_runs_in_cycle(&state, task_id, task_ctx.current_cycle).await?;
if pending_items > 0 {
    // 有 items 未被处理，不应结束 cycle
    continue 'cycle;
}
```

#### 方案 C: SQLite 写入弹性

若根因为 SQLite busy timeout，增加 retry 和 backoff：

```rust
// 在 heartbeat/event 写入处
retry_sqlite_write(|| insert_event(...)).await?;
```

### 2.3 推荐路径

1. 先实现方案 A（防御性检查），使问题从静默成功变为显式失败
2. 通过 2.1 调查确认根因
3. 根据根因选择方案 B 或 C
