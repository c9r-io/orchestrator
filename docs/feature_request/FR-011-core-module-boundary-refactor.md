# FR-011 validate/scheduler/runner 职责拆分与验证逻辑去重

- ID: `FR-011`
- Priority: `P1`
- Status: `Proposed`
- Owner: `orchestrator core`

## 背景

`core` 已具备较强的工作流编排能力，但若按长期维护标准审视，`validate.rs`、`scheduler.rs`、`runner.rs` 已形成明显的“大内核文件”与“高概念密度”问题。当前代码已能工作，但复杂度增长速度正在接近治理阈值。

这类技术债不会立刻造成线上故障，但会持续降低以下能力：

- 新功能进入主线的速度
- 变更影响面的可判断性
- 测试定位效率
- 安全与执行语义的可审计性

## 问题陈述

- 验证逻辑存在多处重复分支和相近实现，修改约束时容易出现不一致。
- 调度、执行、校验、运行前检查等职责在大文件中高度耦合，阅读门槛高。
- 当前代码结构已经不利于继续引入更严格的控制面、安全与 SecretStore 生命周期治理。

## 目标

- 将 `validate.rs`、`scheduler.rs`、`runner.rs` 拆分为职责清晰、边界稳定的模块集合。
- 消灭重复验证逻辑，形成单一权威入口。
- 保持现有行为、CLI 合同、测试语义不回归。
- 为后续安全治理、沙箱治理、执行器扩展预留稳定边界。

## 非目标

- 不在本 FR 中重写整个调度器架构。
- 不要求引入全新工作流 DSL 或新的执行模型。
- 不以“减少代码行数”作为目标；重点是边界清晰和重复消除。

## 范围

### In

- `core/src/config_load/validate.rs`
- `core/src/scheduler.rs` 及其直接耦合的内核逻辑
- `core/src/runner.rs`
- 相关测试的分层整理与命名对齐

### Out

- CLI UX 重构
- 数据库 schema 改造
- protocol / RPC 层重构

## 需求

### 1. 验证逻辑单一化

- 工作流验证必须只有一个主入口负责聚合规则。
- project-scoped 与 agent-ref-scoped 的公共规则不得继续复制粘贴维护。
- 动态步骤、prehook、execution profile、自引用安全等规则应拆分为独立校验单元，由统一入口编排。

### 2. 调度职责拆分

- `scheduler` 应至少在以下职责上形成清晰分层：
  - 任务生命周期协调
  - cycle/segment 驱动
  - phase/item 执行编排
  - 查询/观察接口
  - 安全与回滚协调
- 各层之间应通过稳定数据结构交互，避免横向读写隐式状态。

### 3. runner 职责拆分

- `runner` 中的以下关注点应分离：
  - runner policy 校验
  - execution profile 解析
  - sandbox backend 探测
  - 实际进程 spawn
  - 输出脱敏与采集接线
- 平台差异逻辑不得继续散落在通用执行路径中。

### 4. 测试与回归边界

- 拆分过程中必须保留现有行为语义，优先重用现有测试。
- 对重复逻辑收口后，应新增回归用例，证明不同入口共享同一套规则。

## 设计约束

- 不得通过引入新的“超级 util 模块”掩盖耦合。
- 新模块命名应表达职责，而非表达历史来源。
- 优先抽离纯函数与纯规则，再处理编排层重组。

## 验收标准

- `validate.rs`、`scheduler.rs`、`runner.rs` 的核心职责完成可解释拆分。
- 公共验证规则不再存在并行重复实现。
- 相关测试全部通过，且新增针对“单一验证入口”的回归测试。
- 代码评审可以明确指出每个新模块负责的唯一主职责。

## 风险与缓解

- 风险：大规模文件移动会放大 review 成本。
  - 缓解：按验证、runner、scheduler 三阶段拆分，避免一次性巨变。
- 风险：拆分后可能出现隐式行为偏移。
  - 缓解：优先保持 public API 不变，依赖现有测试集锁定行为。
- 风险：去重过度导致抽象层级过高。
  - 缓解：先抽规则，再抽流程，拒绝为了“漂亮架构”牺牲可读性。
