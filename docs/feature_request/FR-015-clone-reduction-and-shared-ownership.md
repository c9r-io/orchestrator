# FR-015 高频 `clone()` 优化与共享所有权治理

- ID: `FR-015`
- Priority: `P2`
- Status: `Proposed`
- Owner: `orchestrator core`

## 背景

随着调度、配置、事件和控制面功能扩张，系统中存在较多 `String`、配置对象和中间状态的显式克隆。对于当前单机场景，这些克隆未必立刻造成严重性能问题；但在高频 RPC、任务循环、事件流和日志路径上，重复分配会逐步侵蚀吞吐与延迟表现，也会放大内存占用。

本 FR 的核心不是“机械消灭所有 clone”，而是识别热路径中不必要的所有权复制，并用更稳定的共享策略替代。

## 问题陈述

- 高频 `clone()` 会增加分配和拷贝成本。
- 大对象克隆会降低热点路径可预测性。
- 当前代码中部分克隆是为了绕过借用限制，而不是出于真实语义需要。

## 目标

- 识别高频热路径中的不必要克隆。
- 在合适位置引入 `Arc`、借用、`Cow` 或更轻量的共享表示。
- 在不牺牲可读性和线程安全的前提下，降低重复分配。

## 非目标

- 不以“全仓库零 clone”作为目标。
- 不为微小收益引入复杂生命周期技巧。
- 不在本 FR 中做大规模性能重写。

## 范围

### In

- 任务调度主路径
- gRPC 请求/响应构造热点
- 事件与审计热点
- 配置与执行上下文的重复传递

### Out

- 测试代码中的便捷克隆
- 一次性冷路径中的低价值优化

## 需求

### 1. 热路径优先

- 必须先通过 profile、代码审查或指标识别高频克隆热点，而不是盲目全仓库替换。
- 优先治理：
  - 高频 `String` 克隆
  - 大配置对象重复克隆
  - 为跨任务共享而反复复制的只读数据

### 2. 共享策略约束

- 对长期共享只读对象，优先考虑 `Arc`。
- 对“借用优先、必要时拥有”的文本或小对象，优先考虑 `Cow` 或借用接口。
- 不得为了消灭一次克隆而引入明显更复杂、更脆弱的生命周期结构。

### 3. 可测性

- 若某类优化针对已知热点，必须能通过基准、采样或计数证明收益。
- 至少应保留一套可重复的性能观察方式，而不是纯主观优化。

## 验收标准

- 热路径克隆热点形成清单。
- 至少一批高价值热点完成共享化治理。
- 相关路径性能或分配行为有可验证改善。
- 代码评审能清楚解释为什么该处使用 `Arc`/`Cow`/借用，而不是单纯风格偏好。

## 风险与缓解

- 风险：为避免克隆引入过度复杂的借用关系。
  - 缓解：将可维护性置于微优化之前。
- 风险：滥用 `Arc` 导致共享语义模糊。
  - 缓解：仅对跨任务、跨请求、只读共享数据使用 `Arc`。
- 风险：没有基准支撑，优化收益不可见。
  - 缓解：要求热点证据和前后对比。
