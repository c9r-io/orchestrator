# FR-010 控制面安全基线收紧与强制 mTLS

## 背景

当前控制面已经具备 TCP + TLS、客户端证书、角色鉴权与审计能力，但仍存在两个不满足最高安全基线的缺口：

1. daemon 仍暴露 `--insecure-bind` 启动路径，容易被误用于非本机或半生产环境。
2. 安全 TCP 模式当前允许可选客户端证书握手，握手层不是强制 mTLS，认证失败更多依赖 RPC 层拒绝。

在本项目逐步向更稳定、更可复用的控制面演进时，这两个缺口会削弱可信边界，也会放大误配置风险。

## 问题陈述

- 仅靠日志警告不足以阻止 `--insecure-bind` 被错误用于长期运行实例。
- “TLS + 应用层鉴权” 不等价于 “强制 mTLS”；未携带客户端证书的连接仍会完成部分传输层协商。
- 当前安全基线对“本机开发便利性”让步过多，不利于形成默认安全姿态。

## 目标

- 默认关闭 `insecure-bind`，并在标准构建产物中移除或显式禁用该能力。
- 将安全 TCP 控制面升级为握手层强制 mTLS，未携带合法客户端证书的连接不得进入 RPC 层。
- 明确区分“本机 UDS 模式”和“远程控制面模式”的安全契约。
- 保留可审计、可回溯的控制面接入记录。

## 非目标

- 不在本 FR 中引入 OIDC、SSO、外部 IAM 或多集群联邦鉴权。
- 不在本 FR 中替换现有 UDS 本地通信模型。
- 不要求一次性重写现有控制面全部证书自举逻辑。

## 范围

### In

- `orchestratord` 启动参数、构建选项、帮助文档与错误提示
- 安全 TCP 模式下的 TLS/mTLS 握手策略
- 控制面审计字段补充与拒绝路径观测
- CLI/文档中关于控制面暴露面的默认建议

### Out

- 控制面协议重构
- 全量 RBAC 模型扩张
- 外部 CA / KMS 托管

## 现状证据

- daemon 仍支持 `--insecure-bind` 明文 TCP 启动路径
- 安全 TCP 模式当前为 `client_auth_optional(true)`，不是强制双向认证
- 现有 FR-002 已解决“有安全能力”，但尚未收紧到“默认不可误用”

## 需求

### 1. `insecure-bind` 能力收紧

- 正式发布构建中不得默认提供 `--insecure-bind`。
- 若保留该入口，必须通过显式构建特性或开发模式开关启用，且默认关闭。
- 当用户尝试在未启用相应构建特性的二进制中使用 `--insecure-bind` 时，必须得到明确、可执行的错误提示。

### 2. 强制 mTLS 握手

- 安全 TCP 模式下，服务端必须要求客户端证书参与握手。
- 未提供证书、证书格式无效、证书不受信任的连接，必须在握手阶段直接失败。
- 应用层鉴权仍保留，但仅用于“已完成 mTLS 的主体授权”，不再承担兜底认证职责。

### 3. 审计与可观测性补强

- 必须区分以下拒绝类型：
  - 握手前拒绝
  - 证书校验失败
  - 主体不存在
  - 主体被禁用
  - 角色权限不足
- 对管理面操作保留结构化审计线索，至少覆盖时间、来源、RPC、主体、鉴权结论、原因码。

### 4. 文档与运行基线

- 架构文档、README、控制面相关 QA 文档必须同步更新。
- 文档中必须明确：
  - 本地默认推荐 UDS
  - 远程访问必须使用强制 mTLS 的安全 TCP
  - 明文 TCP 仅允许在显式开发构建中使用

## 验收标准

- 默认构建产物中，用户无法直接使用 `--insecure-bind` 启动 daemon。
- 安全 TCP 模式下，无客户端证书连接无法完成握手。
- 已签发但未授权主体、禁用主体、低权限主体分别返回可区分的拒绝结果，并产生审计记录。
- 现有 UDS 模式行为不回归。
- 对应设计文档与 QA 文档完成新增或更新。

## 风险与缓解

- 风险：会打破依赖 `--insecure-bind` 的现有开发脚本。
  - 缓解：提供开发构建特性与迁移说明。
- 风险：强制 mTLS 可能暴露证书初始化流程中的易用性问题。
  - 缓解：保留现有客户端材料签发流程，并增加失败诊断。
- 风险：审计事件扩张后，测试矩阵增大。
  - 缓解：优先以表驱动测试覆盖主要鉴权分支。
