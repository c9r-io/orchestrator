# FR-016 异步上下文锁模型收敛到 `tokio::sync::RwLock`

- ID: `FR-016`
- Priority: `P1`
- Status: `Implemented`
- Owner: `orchestrator core`

## 背景

项目采用 `tokio` 作为异步运行时，核心逻辑中同时存在异步调度、worker、流式查询和控制面 RPC。此类系统若在异步上下文中长期使用阻塞式 `std::sync::RwLock`，就会引入运行时调度阻塞、锁等待不可见、跨 `.await` 使用风险放大等问题。

本 FR 旨在统一异步上下文中的锁模型，减少“同步锁混入 async 主路径”带来的隐性性能与稳定性风险。

## 问题陈述

- `std::sync::RwLock` 在异步任务中可能阻塞执行线程，而不是仅阻塞单个 future。
- 当前若存在同步锁穿过高并发 async 路径，容易造成吞吐下降和尾延迟升高。
- 不同锁语义混用会提高维护者对线程模型和持锁边界的理解成本。

## 目标

- 在异步主路径中优先使用 `tokio::sync::RwLock` 或其他异步友好同步原语。
- 明确哪些状态属于 async 共享状态，哪些仍可保留同步锁。
- 避免阻塞式锁在高并发 async 路径中长期存在。

## 非目标

- 不要求将所有同步锁一刀切替换。
- 不对纯同步、短生命周期、线程外初始化逻辑强制使用 tokio 锁。
- 不在本 FR 中重写所有共享状态模型。

## 范围

### In

- daemon runtime 共享状态
- 调度主路径共享状态
- 控制面查询与观察共享状态
- 任何跨 async 任务读写的全局/长生命周期状态

### Out

- 纯同步初始化阶段临时锁
- 单线程本地变量保护

## 需求

### 1. 锁使用场景梳理

- 必须梳理当前 `RwLock` 的使用场景，并区分：
  - async 主路径共享状态
  - 同步辅助路径状态
  - 只适合消息传递而不适合锁保护的状态

### 2. 异步锁替换原则

- 在 async 主路径上，默认应使用 `tokio::sync::RwLock`。
- 若某共享状态频繁写入或锁粒度过粗，应评估是否应改为更合适的同步原语或消息传递模型。
- 不得将阻塞式锁长期持有并跨越高频 async 调用边界。

### 3. 行为与可观测性

- 替换后不得破坏现有状态一致性语义。
- 对关键共享状态，应明确持锁边界和禁止跨 `.await` 的约束。
- 必要时增加锁竞争或运行态观测，帮助识别替换收益。

## 验收标准

- async 主路径中的阻塞式 `std::sync::RwLock` 使用完成梳理并收敛。
- 应替换的共享状态改为 `tokio::sync::RwLock` 或等价异步友好机制。
- 对保留同步锁的场景有明确文档理由。
- 关键路径测试通过，且无已知状态一致性回归。

## 风险与缓解

- 风险：机械替换锁类型导致 borrow 和持锁边界变化，引入新 bug。
  - 缓解：按状态域逐步替换，并辅以并发测试。
- 风险：部分场景实际上更适合 channel/actor，而不是 RwLock。
  - 缓解：梳理阶段先判断状态模型，再决定是否替换为异步锁。
- 风险：tokio 锁被误持有过久，仍影响吞吐。
  - 缓解：在设计文档中明确持锁约束和推荐模式。
