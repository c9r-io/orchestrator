# FR-052: Inflight Wait Heartbeat-Aware Timeout

- **优先级**: P1
- **状态**: Proposed
- **关联**: FR-038 (Design Doc 50), Design Doc 64
- **日期**: 2026-03-15

---

## 背景

FR-038 引入的 `wait_for_inflight_runs()` 使用硬编码 120 秒超时，不感知 heartbeat。在大规模场景中（130 items, max_parallel=4），正常运行的 agent 子进程因超时被误判为 orphan，导致 task 过早失败（仅完成 26/130 items）。

同时，超时后 daemon 不对 orphan 子进程执行 kill 或 waitpid，导致 zombie 进程堆积、DB 状态永久不一致。

---

## 需求

### R1: Heartbeat 感知的超时逻辑

`wait_for_inflight_runs()` 应查询 in-flight runs 的最近 heartbeat 事件。若任一 run 的 heartbeat 在 grace period 内，视为活跃，重置超时计时器。

### R2: 可配置超时参数

在 `safety` 配置中新增：
- `inflight_wait_timeout_secs`（默认 300s）— 无 heartbeat 活动时的最大等待秒数
- `inflight_heartbeat_grace_secs`（默认 60s）— heartbeat 间隔超过此值视为 stale

旧 YAML 未配置时使用 serde default，无需修改。

### R3: 超时后子进程清理

`inflight_wait_timeout` 触发后，对 remaining runs 执行：
1. SIGTERM → grace period → SIGKILL
2. 更新 DB 中 exit_code 和 ended_at
3. waitpid 回收 zombie 进程

### R4: 增强的超时诊断事件

超时事件应包含：elapsed_secs、since_last_activity_secs、remaining_runs 数量、remaining item IDs、PIDs、has_recent_heartbeat。

---

## 验收标准

- [ ] AC1: `wait_for_inflight_runs()` 在 in-flight run 有活跃 heartbeat 时持续等待，不触发超时
- [ ] AC2: `safety.inflight_wait_timeout_secs` 和 `safety.inflight_heartbeat_grace_secs` 可在 workflow YAML 中配置，未配置时使用默认值
- [ ] AC3: 超时后 daemon 对 remaining runs 执行 SIGTERM/SIGKILL，更新 DB exit_code 和 ended_at，回收 zombie
- [ ] AC4: 超时事件包含完整诊断信息（elapsed、last_activity、remaining items、PIDs、heartbeat 状态）
- [ ] AC5: 130 items / max_parallel=4 的 full-qa 场景不再因 inflight_wait_timeout 过早失败
- [ ] AC6: 无 heartbeat 的 orphan runs 在 timeout 后仍能正确超时并清理

---

## 风险

| 风险 | 缓解 |
|------|------|
| Heartbeat 持续重置导致永不超时 | Stall detection (FR-045) 会在 30 个 stagnant heartbeats 后 kill 进程，两者互补 |
| SIGKILL 后 DB 状态不一致 | kill 后立即更新 DB exit_code=-9, ended_at=now |
| 旧 YAML 兼容性 | serde(default) 确保向后兼容 |
