## 背景

当前 SecretStore 已具备本地密钥文件、AEAD 加密、密文 envelope 与权限校验能力，已经明显优于纯明文配置。但从正式安全治理标准看，它仍停留在“单主密钥 + 本地恢复保护”的阶段，尚未形成完整的密钥生命周期治理。

缺失的能力主要包括：

- 正式轮换
- 吊销与失效治理
- 密钥使用与管理审计

这意味着系统对“密钥长期使用风险”“密钥泄露后的控制能力”“合规级可追溯性”仍不足。

## 问题陈述

- 当前密钥模型默认长期复用，缺少轮换窗口与迁移机制。
- 一旦密钥泄露或主体变更，缺少正式吊销与重新加密流程。
- 审计链更多关注资源访问结果，缺少围绕密钥生命周期的独立治理记录。

## 目标

- 为 SecretStore 建立正式的密钥生命周期模型。
- 支持安全、可回滚的密钥轮换流程。
- 支持密钥吊销、禁用、过期与状态追踪。
- 对密钥生成、激活、轮换、吊销、解密失败、迁移结果形成独立审计链。

## 非目标

- 不在本 FR 中接入外部 KMS/HSM。
- 不要求一次性支持多租户远程密钥托管。
- 不以“完全隐藏本地密钥文件”作为目标；重点是生命周期治理。

## 范围

### In

- SecretStore 主密钥元数据模型
- 密钥状态机与轮换流程
- 历史密钥兼容读取与受控淘汰
- 密钥治理审计事件
- 运维/CLI 可见性

### Out

- 第三方 KMS 集成
- 非 SecretStore 资源的统一加密框架

## 需求

### 1. 密钥状态模型

- 系统必须至少支持以下密钥状态：
  - `active`
  - `decrypt_only`
  - `revoked`
  - `retired`
- 任一时刻只能有一个 `active` 写入密钥。
- 被标记为 `decrypt_only` 的密钥可以用于历史密文读取，但不得继续用于新写入。

### 2. 正式轮换

- 系统必须支持创建新主密钥并切换为 `active`。
- 轮换期间应允许旧密钥继续解密历史数据，直到迁移完成。
- 系统必须支持 SecretStore 数据重加密或按读取渐进迁移的明确策略，且行为应可配置。
- 轮换结果必须可观测，包括成功、部分失败、剩余待迁移资源数量。

### 3. 吊销与失效治理

- 系统必须支持将指定密钥标记为 `revoked`。
- 被吊销密钥不得继续参与新写入；是否允许历史解密必须有明确策略并留下审计记录。
- 对“活跃密钥直接吊销”这类高风险操作必须给出明确保护或双确认机制。

### 4. 审计链

- 以下行为必须产生独立审计事件：
  - 密钥创建
  - 密钥激活
  - 密钥轮换开始
  - 密钥轮换完成
  - 密钥吊销
  - 解密失败
  - 发现缺失/损坏密钥元数据
- 审计事件至少应包含：时间、操作类型、密钥标识、发起主体或来源、结果、原因码。

### 5. 可运维性

- 系统必须提供查看当前活跃密钥、历史密钥状态、最后轮换时间的能力。
- 对于“密钥缺失但数据库中仍有受该密钥保护的数据”这类异常场景，必须给出清晰诊断。
- 文档必须说明灾难恢复和轮换失败回滚路径。

## 验收标准

- SecretStore 支持正式轮换并可验证新旧密钥状态切换。
- 历史密文在轮换窗口内可由旧密钥解密，新写入统一使用新活跃密钥。
- 吊销操作可追踪、可审计，并阻止不安全继续写入。
- 缺失密钥、损坏元数据、使用被吊销密钥等场景都有清晰错误与审计记录。
- 文档与 QA 覆盖主路径、失败路径、恢复路径。

## 风险与缓解

- 风险：轮换涉及历史密文迁移，失败处理复杂。
  - 缓解：先建立状态模型与审计，再实现分阶段迁移。
- 风险：本地文件密钥与数据库元数据可能失配。
  - 缓解：引入启动期一致性检查与修复指南。
- 风险：错误吊销活跃密钥可能导致 SecretStore 不可用。
  - 缓解：对高风险操作增加显式保护和恢复指引。
