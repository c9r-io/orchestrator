# FR-037: Dynamic Items 触发的循环溢出 — max_cycles 约束失效

**优先级**: P1
**状态**: Proposed
**发现来源**: `docs/plan/follow-logs-callback-execution.md` 测试执行监控
**关联 Ticket**: `docs/ticket/20260313-cycle-overflow-dynamic-items.md`
**创建日期**: 2026-03-13

---

## 1. 问题描述

### 1.1 现象

在 self-bootstrap 工作流（`mode: fixed`, `max_cycles: 2`）执行过程中，task 的 cycle 计数器突破了 `max_cycles=2` 的限制，一路攀升到 cycle 10，最终被 FR-035 引入的 L2 退化循环熔断机制（`degenerate_cycle_detected`）兜底拦停。

时间线：

| 时间 | 事件 | 说明 |
|------|------|------|
| 14:05:48 | cycle_started cycle=1 | 正常 |
| 14:12:13 | cycle_started cycle=2 | 正常，qa_doc_gen 产出 3 个 dynamic items |
| 14:18:19 | cycle_started cycle=3 | **异常**，max_cycles=2 应已阻止 |
| 14:18:44 | cycle_started cycle=4 | 异常，间隔 25s |
| 14:18:46 | cycle_started cycle=5 | 异常，间隔 2s |
| ... | cycle 6-9 | daemon 重启后继续溢出 |
| 14:30:31 | cycle_started cycle=10 | L2 熔断触发，task paused |

### 1.2 触发条件

1. 工作流配置 `mode: fixed`, `max_cycles: 2`
2. Cycle 1 的 `qa_doc_gen` 步骤输出 `regression_targets`，触发 `generate_items` 创建 3 个 `source=dynamic` 的 task items
3. Cycle 2 的 self_test 失败（cargo check/test 不通过）
4. Dynamic items 的 `qa_testing` 步骤与 task-scoped 步骤并行推进

### 1.3 影响

- **API token 浪费**：cycle 3-10 的 implement 步骤各消耗一次 LLM 调用，其中 cycle 3-5 在 daemon shutdown 前仅运行数秒即被中断
- **数据污染**：多余 cycle 产生的 checkpoint、binary_snapshot、events 堆积在数据库中
- **依赖 L2 兜底**：max_cycles 约束名存实亡，安全性完全依赖 FR-035 的退化循环检测，而该检测需要 4+ 个 cycle 才能触发

## 2. 根因分析

### 2.1 Cycle 推进逻辑

在 `core/src/scheduler/loop_engine/mod.rs` 中，cycle 计数器在循环迭代开始时**无条件递增**：

```rust
task_ctx.current_cycle += 1;  // line ~126
```

`max_cycles` 的检查发生在 cycle 执行**完成后**，通过 `evaluate_loop_continuation()` 调用 `evaluate_loop_guard_rules()`（`continuation.rs`）：

```rust
// continuation.rs line ~173
LoopMode::Fixed => {
    let max = loop_policy.guard.max_cycles.unwrap_or(1);
    if current_cycle >= max {
        Some((false, "fixed_cycles_complete".to_string()))
    } else {
        Some((true, "fixed_cycle_continue".to_string()))
    }
}
```

### 2.2 Dynamic Items 打破了"一个 cycle 一次完整 pipeline"的假设

`qa_doc_gen` 通过 `generate_items` post-action 在 Cycle 1 结束时创建 3 个 dynamic items。这些 items 以 `status=pending` 进入 task_items 表。

在 `segment.rs` 中（lines ~393-405），当存在 dynamic items 时，items 列表被过滤为**仅包含 dynamic items**：

```rust
let has_dynamic = all_items.iter().any(|i| i.source == "dynamic");
let mut items: Vec<_> = if has_dynamic {
    all_items.into_iter().filter(|i| i.source == "dynamic").collect()
} else {
    all_items
};
```

问题在于：每个 dynamic item 完成其 pipeline 段（如 qa_testing 结束）后，loop engine 会重新评估是否需要新 cycle。由于 self_test 持续失败，unresolved items 始终存在，而 `Fixed` 模式的 `evaluate_loop_guard_rules` 中 `_unresolved` 参数被忽略——cycle 计数器已在开头递增，但 continuation 检查只在 segment 执行完毕后才生效，dynamic items 的并行完成在两者之间制造了"窗口"，导致新 cycle 在 continuation 检查前就被触发。

### 2.3 FR-035 L2 为何未能早期拦截

L2 退化循环检测的触发条件是：
- `current_cycle >= 4`
- 最近 4 个 `cycle_started` 的 3 个间隔**全部**低于 `min_cycle_interval_secs`（默认 60s）

Cycle 1-2 间隔约 6 分钟（正常），所以直到 cycle 4 开始检测时，仍有一个长间隔，需要等到 cycle 7+ 才能凑齐 3 个短间隔。最终在 cycle 10 才满足条件。这是**设计上的延迟**，而非 L2 本身的 bug。

## 3. 目标

### 3.1 Goals

1. **Proactive max_cycles 约束**：在 cycle 递增前检查 `max_cycles`，使 `mode: fixed, max_cycles: 2` 的 task 在完成 2 个完整 cycle 后立即停止，不进入 cycle 3
2. **Dynamic items 不额外计 cycle**：dynamic items 的 pipeline 处理应在其所属 cycle 内完成，不应触发新 cycle 的推进
3. **向后兼容**：不改变 `LoopMode::Infinite` 和 `LoopMode::Once` 的行为，不影响 FR-035 L1/L2 机制

### 3.2 Non-goals

- 不重新设计 dynamic items 的 generate_items 机制
- 不修改 qa_doc_gen 的输出格式或 regression_targets 语义
- 不改变 `loop_guard` step 的定位（它仍然是 pipeline 末端的收口步骤）

---

## 4. 设计方案

### 4.1 在 cycle 递增前增加 max_cycles 前置检查

在 `mod.rs` 的循环迭代顶部，`current_cycle += 1` **之前**插入 proactive 检查：

```rust
// Before incrementing cycle counter
if let LoopMode::Fixed = loop_policy.mode {
    let max = loop_policy.guard.max_cycles.unwrap_or(1);
    if task_ctx.current_cycle >= max {
        // Emit event and break
        emit_event(state, task_id, "max_cycles_enforced", json!({
            "current_cycle": task_ctx.current_cycle,
            "max_cycles": max,
        })).await?;
        break;
    }
}
task_ctx.current_cycle += 1;
```

这确保了即使 continuation 检查因 dynamic items 并行而被跳过，cycle 也不会超过 max_cycles。

### 4.2 Dynamic items 的 cycle 归属

Dynamic items 应被视为其**创建时所在 cycle** 的一部分。在 `item_generate.rs` 创建 dynamic item 时，记录 `created_at_cycle`：

```rust
// In item_generate.rs, when creating dynamic items
item.created_at_cycle = task_ctx.current_cycle;
```

在 segment 执行时，仅处理 `created_at_cycle <= current_cycle` 的 dynamic items，防止后续 cycle 的 dynamic items 被提前处理。

### 4.3 Cycle boundary fence

在 `segment.rs` 的 task segment 完成后、下一次循环迭代前，增加一个显式的 cycle boundary 检查：

```rust
// After segment completion, before next iteration
let should_continue = evaluate_loop_guard_rules(
    &loop_policy, task_ctx.current_cycle, unresolved
);
if let Some((false, reason)) = should_continue {
    emit_event(state, task_id, "loop_terminated", json!({
        "reason": reason,
        "cycle": task_ctx.current_cycle,
    })).await?;
    break;
}
```

这提供了双重保障：即使 4.1 的前置检查被绕过（例如由于并发 segment 执行），boundary fence 仍能阻止新 cycle。

## 6. 测试策略

### 6.1 单元测试

1. `evaluate_loop_guard_rules` 在 `current_cycle == max_cycles` 时返回 `(false, "fixed_cycles_complete")`
2. `evaluate_loop_guard_rules` 在 `current_cycle > max_cycles` 时同样返回 `false`（防御性）
3. Proactive cycle check 在 `current_cycle >= max_cycles` 时阻止递增

### 6.2 集成测试

1. 创建 `mode: fixed, max_cycles: 2` 的 workflow，其中 qa_doc_gen 产出 dynamic items，验证 task 在 2 个 cycle 后停止
2. 验证 dynamic items 仍然被正确处理（qa_testing 执行目标文件），不因 cycle 限制而跳过

### 6.3 回归测试

1. FR-035 的 L1/L2 机制不受影响
2. `LoopMode::Infinite` 的正常多 cycle 行为不受影响
3. 无 dynamic items 的 `mode: fixed` workflow 行为不变

## 8. 验收标准

1. `mode: fixed, max_cycles: 2` 的 task 在完成 2 个 cycle 后停止，cycle 计数器不超过 2
2. Dynamic items 的 qa_testing 在 cycle 2 内完成或被标记为 `pending`（不触发新 cycle）
3. `degenerate_cycle_detected` 事件在正常 2-cycle 执行中不触发
4. 现有 self-bootstrap 测试计划（`follow-logs-callback-execution.md`）可在不触发 cycle 溢出的情况下完成
5. `cargo test --workspace --lib` 通过
